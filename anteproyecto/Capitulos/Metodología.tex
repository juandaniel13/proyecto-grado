\chapter{Metodología}

El proyecto se desarrollará bajo un esquema ágil, adaptando la metodología XP a
las necesidades del proyecto y del equipo de desarrollo. La elección de XP se
justifica por su gran flexibilidad, la cual permite ajustar el proceso de
desarrollo sin fricción ante cambios en los requisitos o en la dinámica del
equipo. Esta flexibilidad es fundamental para asegurar la satisfacción del
cliente, ya que facilita la entrega continua de valor, la integración inmediata
de su retroalimentación y la corrección oportuna de desviaciones entre lo
solicitado y lo implementado. Además, XP se adapta sin dificultad a equipos
pequeños como el de este proyecto, donde la comunicación directa con el cliente
y la ausencia de roles rígidos agilizan aún más la respuesta a sus necesidades.

En este caso, no se considerará el rol del gerente; la comunicación e
integración entre el cliente y el equipo de desarrollo será directa. Sobre este
último recaen las actividades de análisis, diseño, pruebas y programación de la
aplicación web.
\section{Fase 1: Exploración}

\subsection{Definición de requerimientos funcionales} En esta primera  etapa  se
documentan, junto con el cliente, los requerimientos funcionales del  sistema  a
través de  historias  de  usuario.  Cada  historia  de  usuario  representa  una
funcionalidad que será añadida a la aplicación web.

\subsection{Definición de la arquitectura del sistema y requerimientos no
    funcionales} En esta etapa se especifican los requerimientos no  funcionales
del
sistema en un formato claro  y  se  define  una  arquitectura  de  software  que
satisfaga dichos  requerimientos  y  que  sea  mantenible  para  el  proceso  de
desarrollo iterativo e incremental.

\subsection{Selección de tecnologías} En esta etapa se evalúan y seleccionan las
tecnologías más adecuadas para construir la aplicación web, teniendo  en  cuenta
el nivel de conocimiento del equipo  de  desarrollo  y  su  idoneidad  para  las
funcionalidades a implementar.

\subsection{Artefactos} En esta fase, el artefacto entregado serán las historias
de usuario.

\section{Fase 2: Planificación de la iteración}

\subsection{Priorización de historias de usuario} En esta  etapa  se  reúnen  el
equipo de desarrollo con el cliente, y  este  último  indica  las  historias  de
usuario que serán incluidas en la siguiente iteración.

\subsection{Estimación del esfuerzo por cada historia de usuario} En esta etapa,
el equipo de desarrollo estima el esfuerzo requerido para completar cada una  de
las historias de usuario priorizadas anteriormente, asignando un número del 1 al
13 basado en la sucesión de Fibonacci. Cuanto mayor sea el número, mayor  es  el
esfuerzo necesario para completar la historia de usuario.

\subsection{Asignación de tareas} Cada historia  de  usuario  se  descompone  en
tareas más sencillas y manejables. Cada  integrante  del  equipo  de  desarrollo
selecciona aquellas en las que desee trabajar. Estas tareas  se  representan  de
forma visual en un tablero Kanban virtual para hacer seguimiento del avance.

\subsection{Cronograma para la iteración} En esta etapa se define el  cronograma
de la iteración, que  contiene  el  orden  en  que  se  completarán  las  tareas
asignadas.

\subsection{Artefactos} En esta  fase,  los  artefactos  a  entregar  serán  las
historias de usuario priorizadas y estimadas para la iteración actual, junto con
el cronograma de la iteración.

\section{Fase 3: Iteración}

\subsection{Sesión rápida de diseño} El objetivo de esta etapa  es  eliminar  la
incertidumbre  sobre  cómo  se  implementarán  las  tareas  asignadas   a   cada
desarrollador. Para ello, el equipo de desarrollo se reúne y dibuja  rápidamente
los diagramas necesarios (sin profundizar  demasiado  en  el  detalle,  pues  se
refinarán más adelante) para aclarar las ideas en cuanto al diseño del código  y
los flujos de información.

\subsection{Implementación} El desarrollo se llevará a cabo mediante un  enfoque
basado en pruebas (TDD, por sus siglas en inglés).  Primero  se  escribirán  las
pruebas unitarias y luego se implementará el código para que estas  se  cumplan.
Finalmente, se integrará todo el trabajo en un repositorio mediante  integración
continua, ejecutando todas las pruebas para asegurarse de que no haya errores.

También  se  ejecutarán  pruebas  de  integración  para  validar   el   correcto
comportamiento de los distintos componentes de la aplicación.

Cuando se terminen  de  programar  las  funcionalidades,  se  implementarán  las
pruebas  de  aceptación  en  concordancia  con  los  criterios   de   aceptación
especificados en las historias de usuario.

\subsection{Validación con el usuario} El equipo de desarrollo se reúne  con  el
cliente y obtiene la retroalimentación de las funcionalidades implementadas,  si
hay funcionalidades a corregir, se organizan en forma de tareas y se desarrollan
en la siguiente iteración, la retroalimentación del cliente se documentará en un
acta de reunión.

\subsection{Artefactos} El artefacto entregado en esta fase es el
acta de reunión con el cliente.

\section{Fase 4: Despliegue} En esta fase, el resultado de la iteración final se
valida con el cliente y posteriormente se despliega cada una de las unidades  de
despliegue que componen la  aplicación  web  (cliente  web,  backend,  bases  de
datos).

\section{Fase 5: Mantenimiento}
\subsection{Detección y corrección de
    errores} En esta etapa, el equipo y el cliente, prueban y valida  el  código
en producción para
detectar  errores  de  funcionamiento  y  problemas  de  rendimiento.   De   ser
encontrados, se corrigen y el código corregido se vuelve a desplegar.

\section{Fase  6:  Muerte}
\subsection{Refinamiento  de  la  documentación  del código} En esta etapa se  refinan  los  diagramas  utilizados  para  diseñar  la
aplicación web. También se construyen los que sean necesarios para asegurar  que
el sistema quede documentado apropiadamente y sea fácil de mantener.

\subsection{Entrega final del producto de software}
En esta etapa se hace entrega de la aplicación web desplegada, el código fuente esta misma en un repositorio público
y la documentación correspondiente (diagramas).

\subsection{Artefactos}
Los artefactos entregados en esta fase son la documentación del sistema, que se compone de los
diagramas UML y C4 que especifican el diseño del software.
